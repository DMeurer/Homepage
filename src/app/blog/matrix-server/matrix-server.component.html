Self-hosting a Matrix server

For a while I was a happy Beeper user. The pitch is great: all your chat networks (WhatsApp, Signal, Telegram and the rest) in a single app. The catch is that every message you send flows through someone else's infrastructure and will be unencrypted at some point. I wanted that same one-inbox-for-everything experience, but on hardware I control and with data that is mine.

Matrix makes that possible. It is an open, federated chat protocol, and crucially it has a whole ecosystem of bridges that connect it to the closed networks everyone else uses. This is actually how Beeper works in the first place. So the plan is simple: build a self-hosted Beeper replacement by running a Synapse server with all the bridges I want.

Not reinventing the wheel

I did not set up Synapse, Postgres, a reverse proxy and a dozen bridges by hand. That looked like something I really dont want to do and kept me from starting with this a long time. Instead the whole server is described by Slavi Pantaleev's excellent matrix-docker-ansible-deploy playbook. Everything runs as Docker containers, and a single Ansible run brings the server from "fresh VPS" to "working matrix server".

The entire configuration lives in one vars.yml file. The core of it is almost boring: pick a domain, pick a homeserver implementation (Synapse, the reference server), and let the playbook manage its own Traefik reverse proxy, which fetches Let's Encrypt certificates on demand.

The bridges

This is where the playbook earns its keep. Each network I want to fold into Matrix is essentially a single boolean in the config. Behind each one is a mautrix (or similar) appservice that logs into the other network on my behalf and mirrors the conversations into Matrix rooms.

That gives me WhatsApp, Signal, Telegram, Discord, Instagram, Google Messages, LinkedIn and even Steam, all landing in the same client. The bridged rooms are end-to-end encrypted, so the messages aren't sitting around in plaintext just because they took a detour through my server. This is very important for me, because with only limited control over this installation, I have to fully trust the editors of the playbook with security. When the messages are encrypted on the server, even if there are vulnerabilities or I miss an important security update, no one will be able to read my or the bridged messages.

Relay mode

This was very much beyond my initial use case and idea, but with unlimited possibilities on my own chat network come many stupid ideas.
But lets take a step back. Relay mode is intended to connect users in a single group chat across multiple chat networks. Normally if you want to do this, you will use a bot on both networks to relay messages. For Matrix this done extremely well with one bot per user on the other network, but on the other network its usually the bot account writing "Alice sent a message: [message]". Sadly WhatsApp wont give out bot account easily, so I had to use my own account.

The uncomfortable hosting truth

Here is the part that slightly undercuts the whole "own my data" motivation: the server currently runs on a Hetzner VM, not in my own homelab. My homelab server broke not too long ago, and a chat server that I depend on needs to be reliable more than it needs to be ideologically pure. A rented machine that stays up, my own didn't. I am at peace with the trade-off. For now.

From "just me" to a few friends

The server was only ever meant for me. But a homeserver that already bridges half the internet turns out to be a tempting thing to share, and a couple of friends wanted to join. That changed the calculus a bit: a toy for one person can fall over and nobody cares, but once other people rely on it I want to know how it is doing.

Keeping an eye on it with Grafana

Before I gave access to anyone, I added monitoring. I absolutely did cut corners when it came to hardware, renting the cheapest VPS with shared vCPU available, so I need to know when I have to stop allowing more users or need to upscale. The playbook wires up a stack of Prometheus exporters. One for the host, one for Postgres, one for Synapse's own usage stats, and they fee directly into Grafana.

Mostly I use it to watch CPU and memory, so I can answer a single question: can the server take a few more users, or is it already at its limit? It is much nicer to make that call from a graph than from a gut feeling. And it is absolutely possible. The entire stack is running so lightweight, its only using 10 % CPU and 50 % of RAM for one user.
There is only one concern, so big it gets its own chapter

What's next: storage

The VPS has 40 GB of storage, 20 of that already filled with the system and a month worth of my chats. Upscaling the server will double this and the monthly price. So that would be okay only once.
There is one great solution to this problem. The Ansible Playbook actually supports just plonking a S3 Bucket in and using this as media server. And as I highly doubt that I will 20 GB with chat messages, that sounds great. Only thing is, I need a S3 Bucket. Thankfully Hetzner has a ready to use S3 Bucket plan, that will only cost about 5€ per TB, along with some transfer limitations. But most media will be historical and rarely accessed, so I believe the transfer limits will be enough for my usecase.

Now the only thing to worry about is the migration from on-device storage to the S3 Bucket. But that will only become relevant when I run out of space.
