Job Portal

The Job Portal is the first fullstack app I have developed at home. The idea is simple: I want to keep an eye on the job openings of a few companies I'm interested in, without checking a dozen career pages by hand every week. So instead I let a bunch of scrapers do it for me, store every listing in a database, and look at the whole thing through a dashboard.

What makes it more interesting than a one-off scraper is that it doesn't just store the current openings. It takes a snapshot every day, so I can actually see how things change over time: which positions are new, which ones disappeared, and how long a job stays online before it's filled.

The architecture

It's a multi-service application, split into three repositories that are pulled together as git submodules. Each part does one job and talks to the others over HTTP:

  • Backend: a FastAPI service with a PostgreSQL database. It owns all the data and exposes the REST API.
  • Frontend: an Angular dashboard that visualizes the statistics.
  • Runners: the Python scrapers, one per company, running on a schedule.

The data model

The trick to tracking change over time is in how the data is stored. There are three main tables:

  • Companies: the sources I scrape from.
  • Jobs: a single listing, identified by its company and the external job id or URL.
  • Inserts: one row for every day a job was seen during a scrape.

That inserts table is the important one. A job isn't simply "open" or "closed". Instead, every scrape writes down "I saw this job today". By comparing the set of jobs seen on two days I can derive everything else: a job on today's snapshot that wasn't on yesterday's is new, one that was there yesterday but is gone today was removed, and the gap between the first and last time a job was seen tells me how long it was online.

The scrapers

Every company gets its own scraper, and they all push their results to the same API endpoint using a shared client. A small orchestrator runs them one after another, and inside the runners container a cron job kicks the whole thing off once a day. Adding a new source mostly means writing one more scraper class and registering it.

The runners live in a private repository on purpose. The scraping logic is specific to each company's career page, and I'd rather not publish the full list of who I'm watching or how. This is because while some companies allow non-commercial scraping and republishing, other only allow this for personal use. Anything for personal use will stay hidden.
Publicly, the dashboard shows a handful of companies from my region, B. Braun, KLS Matrin and Kauth, while the rest stay hidden.

Data Quality

The data quality of the the produced dataset is by far the worst dataset I have worked with.
Job pages have synonyms for the same thing, and multiple pages have entirely different names for the same stuff. For example: All companies have some sort of open positions for students or apprentices. Some will call this have them together, some split up, some will just call it "after school".
As much as this is really bugging me, I have not implemented a proper cleaning of the scraped data. Definitely not production ready and actually barely ready at all, but I can live with this and fixing this would be a huge amount of work.

Hidden companies and API keys

I wanted the dashboard to be shareable without exposing every source I track. So companies carry a hidden flag, and access is controlled through API keys with different permissions. There are four fixed keys, each with its own scope:

Every read query runs through the same filter: unless the key has the read_hidden permission (or is the admin key), hidden companies are quietly dropped from the results. The frontend ships with the read-only key, so anyone using the public dashboard only ever sees the non-hidden companies. With a full-read key I get the complete picture. The scrapers use a separate write key, which keeps the thing that fills the database away from the keys that only read it.

The API

The backend is where most of the work happens. Besides the basic CRUD it has a fairly rich filtering endpoint: you can query jobs by company, level, location, contract type, by the date they were found, and by their status on that date (new, existing or removed). Titles and functions can even be filtered with regular expressions. The same endpoint can return aggregated statistics instead of raw listings, which is exactly what the dashboard needs.

The dashboard

The frontend is deliberately small. The main page is a chart of the total open positions across all accessible companies over the last 30 days, so I can see trends at a glance. A settings page lets me point it at a different API URL or drop in a stronger API key (to unlock the hidden companies), and everything is stored in the browser's local storage. No backend session, no login. The API key is the login.

The most most information is in the "Time Curve". This diagram will show you the history of all jobs for the company. It will show how many new ones popped up, and how many jobs disappeared.
But the view wont stop at the aggregates. Every single bar is clickable and acts as a filter for a list of jobs below. So can select green bar on a day you want and you will get a list of all jobs created that day.

Deployment

Deployment follows the same homemade pattern I use for my other projects: Docker Compose for the services and a cron job that rebuilds containers when there's a new release. The runners run on their own cron schedule inside the stack, so once it's up it quietly collects a new snapshot every day without me touching it.
